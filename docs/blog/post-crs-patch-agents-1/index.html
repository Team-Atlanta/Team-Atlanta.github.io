As we mentioned in our previous blog post, we enhanced the patching capabilities of Atlantis by ensembling multiple patch agents. In this series of blog posts, we will introduce each of our patch agents in detail and explain the rationale behind their designs.

Diversity for Good

To maximize the effectiveness of ensembling, it is crucial to have diverse agents. If all agents are similar, the ensemble will not perform significantly better than any individual agent. Therefore, we intentionally designed our patch agents to be diverse in their approaches, methodologies, and also models used. We newly developed six patch agents, each with its own unique architecture and motivation, as summarized in the table below.

+---------------+----------------+----------------------------------------------+------------------------------+
| Agent         | Architecture   | Motivation                                   | Used Models                  |
+---------------+----------------+----------------------------------------------+------------------------------+
| Martian       | Workflow       | Simple workflow yet complex tools            | o4-mini, claude-4-sonnet     |
| MultiRetrieval| Agent          | Iterative retrieval and patching             | claude-3.7-sonnet, o4-mini   |
| Prism         | Multi-agent    | Multi-agent system for long-context handling | o4-mini                      |
| Vincent       | Workflow       | Property-based approach for guided patching  | gemini-2.5-pro               |
| Claudelike    | Agent          | Patch generation inspired by Claude code     | claude-3.7-sonnet            |
| Eraser        | Workflow       | Use of custom model for specialized tasks    | Custom model                 |
+---------------+----------------+----------------------------------------------+------------------------------+

Martian: Into the Unknown, Armed with Sophisticated Tools

We first introduce Martian, a patch agent that employs a straightforward workflow but leverages sophisticated tools. The name “Martian” reflects its approach of exploring the unknown, much like a Martian would, by utilizing advanced tools to navigate and understand unfamiliar terrain.

Simple Workflow

Martian’s workflow is straightforward yet effective. It consists of two main steps: fault localization and patch generation. The fault localization step uses crash logs and code exploration to identify the root cause of the bug, while the patch generation step creates a patch to fix the identified issue. A feedback loop ties them together, allowing the agent to refine its understanding based on build failures or test results.

Bug Report  -->  [ Fault Localization ]  -->  [ Patch Generation ]  -->  Patch
                  ^                                                   |
                  |___________________________________________________|
                                   Feedback

The workflow starts with crash logs—typically ASAN reports or stack traces—that provide clues about where things went wrong. During fault localization, the agent explores the codebase iteratively using specialized search tools inspired by CodeRover-S. When a likely root cause is identified, that information flows to patch generation, which focuses narrowly on fixing the identified function.

If the generated patch fails validation—whether it doesn’t compile, breaks existing tests, or doesn’t actually fix the vulnerability—the feedback is filtered and provided back to the localization phase, allowing the agent to refine its understanding and try again.

Sophisticated Tools

What makes Martian unique isn’t its workflow structure, but the sophisticated tools it wields. We designed the agent to simulate how human developers debug and patch software, providing tools that mirror typical debugging workflows.

For fault localization, Martian uses three key code search APIs:

  • search_symbol(symbol_name, path): Finds symbol definitions and returns their source code
  • search_string(string, path): Locates where strings appear in the codebase with line numbers
  • view_file(file_path, offset, limit): Returns source code snippets from specific line ranges

These tools enable the ReAct-based agent to explore codebases systematically, much like a developer following a stack trace through unfamiliar code.

For patch generation, Martian adopts a search/replace approach inspired by Claude Code. Rather than generating entire files or trying to produce correctly formatted git diffs, it focuses on editing a single function at a time:

  • view_function(function_name): Returns a specific function’s source code
  • edit_function(function_name, old_string, new_string): Replaces code segments within that function
  • add_import_module(module_name, file_path): Adds imports when needed (Java only)

This constrained scope reduces context size and helps the LLM focus precisely on the logic that needs fixing. Through extensive testing, we found that Anthropic’s Claude models consistently produced syntactically correct, applicable patches using this approach.

Dealing with Stack Overflow and Timeout Bugs

Beyond general-purpose patching, Martian includes specialized handling for particularly challenging bug classes: stack overflow and timeout errors in Java.

The problem with these bugs is that their crash logs often provide no useful information. Jazzer timeout logs might show only JVM internal threads with no trace of the project’s code. Stack overflow logs display recursive chains of internal JDK functions while truncating the crucial project function that triggered the overflow.

Martian’s solution? Runtime stack tracing. The system periodically captures the JVM’s main thread stack trace every second during bug reproduction. When a crash occurs, the agent uses the last recorded stack trace as a more informative snapshot of what was actually happening in the project code before things went wrong.

This supplementary trace provides a meaningful starting point for fault localization even when the crash log itself is opaque—enabling Martian to generate plausible patches for bugs that would otherwise be impossible to debug automatically.

MultiRetrieval: Context is Everything

The MultiRetrieval agent tackles a fundamental problem in automated patching: how do you give an LLM the right code at the right time? A crash log might point to a specific function, but fixing it requires understanding types, data structures, similar functions, project patterns, and dependencies. Simply dumping entire files overwhelms the LLM with noise. Providing just the crashing function leaves out crucial context.

MultiRetrieval’s answer is augmented retrieval—combining multiple search strategies and enriching retrieved code with contextual information that makes it immediately useful.

Dual Retrieval: Breadth and Precision

When the LLM requests code, MultiRetrieval runs two retrievers in parallel:

Ripgrep provides fast text search, finding all occurrences of a symbol with context (typically 5 lines before and after). This gives breadth—you see everywhere the symbol appears with enough surrounding code to understand usage patterns.

AST-Grep provides structural understanding by parsing code into Abstract Syntax Trees. When searching for a function, it finds the definition specifically, not just any line containing that name. For C/C++, it distinguishes function definitions from type definitions, struct declarations, and preprocessor macros. For Java, it parses fully qualified names like com.example.ClassName.methodName, validates file paths against package structure, and automatically includes relevant import statements.

The combination is powerful: ripgrep catches what AST-Grep might miss in syntactically complex code, while AST-Grep provides precise, structured results when parsing succeeds.

Making Code Self-Documenting

Raw code snippets aren’t enough. MultiRetrieval augments every retrieval with contextual information:

Line numbers accompany every snippet, enabling the LLM to reference precise locations when analyzing or generating patches.

Automatic imports are included for Java methods and classes. Understanding what types are available is crucial to understanding what code does. Consecutive imports are grouped together rather than scattered.

Path rebasing handles mismatches between crash log paths (which use absolute build environment paths) and repository paths. The retriever progressively strips path components until it finds files that actually exist.

These augmentations mean the LLM doesn’t have to guess about details—the code becomes self-documenting.

Priority-Based Filtering

When multiple snippets match a query, returning all of them creates noise. MultiRetrieval assigns priorities:

  • High: Whole-word matches from AST-based retrieval with validated package structure (Java)
  • Medium: Partial-word AST matches
  • Low: Text search fallback and file retrievals

Only the highest-priority tier is kept. If high-priority results exist, medium and low are discarded. A limit of 16 results per query prevents even high-priority matches from overwhelming context.

The principle: better to give the LLM less information that’s highly relevant than more information that’s noisy.

Iterative Exploration

The agent operates through cycles. First, the LLM analyzes the bug report and articulates what it doesn’t know yet. Then it requests code using structured symbol or file queries (3-5 per round). The system retrieves, augments, filters, and delivers results in structured format showing what was searched, where results came from, and the code itself.

After reviewing retrieved code, the LLM decides: request more code, or generate a patch? With improved reasoning models, this decision-making became increasingly reliable.

When patches fail validation, filtered feedback (compilation errors, crash logs, test failures) becomes input for the next iteration. The LLM enters another analysis phase with richer information: the original bug, its previous attempt, why it failed, and all the code it has already explored.

This iterative approach, combined with augmented retrieval that makes every code snippet immediately useful, is why MultiRetrieval effectively handles bugs that require systematic codebase exploration.

Prism: Specialization Through Multi-Agent Teams

Building on MultiRetrieval’s foundation, Prism addresses a specific challenge that emerged during testing: what happens when a bug requires many patch attempts? After several failed iterations, the conversation history grows large—original bug report, previous patches, compilation errors, test failures, and all retrieved code. For hard cases requiring multiple explorations, this quickly consumes tokens.

Prism’s solution? Split the workflow into specialized teams and use notebook-based context compression between attempts.

Three Specialized Teams

Prism divides patching into three teams, each with focused responsibilities:

The Evaluation Team processes crash logs, runs tests, and produces filtered feedback reports. It extracts compilation errors, test failures, and crash information without verbose output.

The Analysis Team has two sub-agents that run sequentially. The Code Context Provider explores the codebase using retrieval strategies similar to MultiRetrieval, but organizes findings into notebook cells—each cell contains a code range with line numbers plus targeted analysis. The Fix Strategy Generator then synthesizes a high-level strategy based on these cells: “Based on cells 3 and 5, follow the pattern of…” This is strategy, not code—guiding what to do rather than how to do it.

The Patch Team generates and reviews patches based on the analysis notebook. The Patch Generator creates line-range replacements following the strategy. The Patch Reviewer validates them: correct replacement? addresses issue? follows strategy? no syntax errors? If validation fails internally, the generator receives feedback and tries again.

A Supervisor routes between teams, managing the iteration cycle until a sound patch emerges or the iteration limit is reached.

Context Compression That Scales

The notebook is the key innovation. Without compression, context grows linearly: original bug + all code from attempt 1 + patch 1 + errors + all code from attempt 2 + patch 2 + test failures…

With Prism’s notebook compression, what carries forward is:

  • Evaluation report (filtered error messages from latest failure)
  • Analysis notebook (5-8 cells with code ranges and insights)
  • Previous patch attempt (for reference)

A patch attempt that explored 20 files compresses into 5-8 cells with targeted insights. When a patch fails, the supervisor cycles back through all three teams. The Analysis Team can add new cells based on the failure while keeping existing ones—no need to re-explore code that’s already in the notebook.

The Trade-offs

This multi-team approach brings benefits and costs:

Token Efficiency: The notebook compresses many code retrievals into focused cells. Context stays manageable across multiple iterations.

Focused Context: Each team gets only what it needs. The Patch Generator works from distilled strategy, not raw error logs.

Specialization: Different teams use different prompting strategies optimized for their specific task.

But there are costs too. Each cycle involves multiple team invocations—worthwhile for complex bugs, but overhead for simple ones. Building a notebook takes time. When the fix is obvious from the crash log, this extra structure slows things down.

Prism complements the ensemble by handling complex bugs that require iterative refinement, while faster agents tackle straightforward cases.

Vincent: When Context Matters More Than Symptoms

During competition testing, we encountered a challenge that stumped our other agents consistently: a seemingly simple double-free bug in nginx. The bug was obvious from the sanitizer report. LLMs correctly identified the root cause. They generated syntactically correct patches that fixed the immediate symptom. Yet these patches were wrong.

This challenge revealed a gap in how LLMs approach patching—and Vincent was our answer.

The nginx Double-Free Challenge

The bug was in ngx_http_process_prefer, a function that handles Prefer headers in HTTP requests. When encountering duplicate headers, it logged an error, freed the previously allocated object, and returned NGX_OK. The problem? It never nullified the pointer after freeing, creating a use-after-free vulnerability when a malicious request contained multiple Prefer headers.

The sanitizer report was crystal clear—double-free at a specific line. The fix seemed obvious: nullify the pointer after freeing. That’s exactly what our agents produced:

ngx_free(r->headers_in.prefer);
r->headers_in.prefer = NULL; // Prevent further misuse
return NGX_OK;

Syntactically correct. Addresses the immediate symptom. Compiles successfully. But wrong.

The problem? nginx has project-specific conventions for handling invalid requests. Header handlers must finalize the HTTP request using ngx_http_finalize_request with an appropriate flag and return NGX_ERROR to indicate failure. Without complying with these rules, the patched program still suffered from another heap overflow caused by the prefer header.

Our agents were fixing symptoms without understanding the project’s semantic requirements. They were doing surface-level repairs when deep context was needed.

Property-Based Approach: Guiding Beyond Symptoms

Vincent’s solution borrows from formal verification: the concept of program properties. A property is a statement about program behavior that should hold true across all executions—like “no null pointer dereference occurs” or “after an INSERT, the database contains the inserted row.”

Vincent adds an intermediate layer that analyzes program properties before generating patches. Interestingly, LLMs already understand this concept with concise prompting:

“In formal terms, a property refers to a condition that must hold true across all possible execution traces…”

For the nginx bug, Vincent extracted properties including:

  • Memory Safety: For any header field, the pointer must be NULL or point to a valid structure.
  • Header Processing Consistency: Single-value HTTP headers must maintain exactly one instance throughout request processing.
  • Error Response Consistency: When encountering invalid headers, the system must either (1) return NGX_ERROR and call ngx_http_finalize_request with NGX_HTTP_BAD_REQUEST or (2) log and return NGX_OK. But never both or neither.

That last property—Error Response Consistency—was what superficial patches missed. With this property in mind, Vincent produced the sound patch:

ngx_free(r->headers_in.prefer);
r->headers_in.prefer = NULL;
ngx_http_finalize_request(r, NGX_HTTP_BAD_REQUEST);
return NGX_ERROR;

This patch doesn’t just fix the symptom. It respects the project’s conventions for error handling, ensuring correct behavior across all code paths.

The property-analysis layer guides LLMs to consider project-specific aspects rather than just observable symptoms. It’s the difference between patching what broke and understanding why it should work.

ClaudeLike: Applying General-Purpose Agent Strategies to Security Patching

When Claude Code launched, we immediately experimented to see how well this general-purpose programming agent handled patch generation. It performed reasonably well when provided with crash logs and context. But integrating it directly into our system wasn’t feasible—we couldn’t modify its internals to add custom tools, and some of its built-in capabilities (like WebFetch and unrestricted Bash) were unnecessary or unpredictable for our controlled patching environment.

Instead of integrating Claude Code, we built ClaudeLike—an agent that adopts its core strategies for security patching.

Two Key Strategies

Through deep analysis of Claude Code’s design, we identified two aspects with significant performance impact:

File editor tools that enable efficient analysis and modification of project structures. Rather than having the agent generate entire files or try to produce git diffs, Claude Code uses targeted operations like reading file ranges, searching with glob/grep patterns, and performing precise string replacements.

Sub-agent delegation for smaller, multi-step tasks. Instead of the main agent executing every operation itself (which lengthens context and distracts from the primary objective), it can spawn a sub-agent to handle specific tasks like “identify which files implement authentication” or “trace how this data structure is initialized.”

These two strategies became ClaudeLike’s foundation.

File Editor Tools: Efficient Agent-Project Interaction

ClaudeLike interacts with projects using six tools inspired by Claude Code:

  • EditTool: Replaces a unique old_string with new_string in a file. The uniqueness requirement ensures precise edits.
  • GlobTool: Returns files matching a glob pattern in a directory.
  • GrepTool: Returns files whose content matches a regex pattern, with optional filename filtering.
  • LSTool: Lists files in a directory.
  • ReplaceTool: Replaces entire file content or creates new files.
  • ViewTool: Returns a range of lines from a file.

Each tool’s description explicitly defines when it should be used, constraints, and valid argument types. By referencing these descriptions, the agent selects appropriate tools to accomplish tasks effectively.

Through repeated benchmarking, we found that performance varied significantly depending on the clarity and specificity of these tool descriptions. Well-defined tools with precise instructions proved essential for effective patch generation.

AgentTool: Keeping Focus on Patching

To analyze a file’s role or locate functionality, agents often need to read multiple files or repeatedly traverse directories. These operations are frequent in patch generation but result in numerous tool calls that lengthen context and may distract from the primary objective—generating a correct patch.

ClaudeLike adopts AgentTool to address this. The main agent can delegate specific tasks to a sub-agent, which operates independently, makes its own tool calls, and returns results. This allows offloading repetitive or verbose tasks (like identifying file functionality or searching for related code), keeping the main agent’s context concise and focused on patching.

To prevent uncontrolled recursion, sub-agents cannot use AgentTool themselves. To avoid unintended modifications, they’re restricted from using EditTool and ReplaceTool—they can search and read, but not modify.

From General-Purpose to Security-Focused

Evaluation results showed that ClaudeLike successfully generated sound patches for most cases in our Round 3 dataset. More importantly, it demonstrated that strategies from general-purpose software engineering agents transfer effectively to security patch generation.

The lesson? Providing well-defined tools with precise instructions and enabling agents to maintain focus by reducing context size are proven strategies—not just for general programming tasks, but specifically for the challenging domain of automated security patching.

Conclusion: Diversity in Action

Each agent we’ve explored brings a distinct approach to automated patching:

Martian combines straightforward workflow structure with sophisticated debugging tools, including specialized runtime tracing for opaque bug classes.

MultiRetrieval solves the information problem through augmented retrieval—dual search strategies, context enrichment, and priority filtering that gives LLMs exactly the right code at the right time.

Prism tackles token constraints through specialization, using multi-agent teams and notebook-based compression to handle complex bugs requiring many iterations.

Vincent looks beyond symptoms to project semantics, using property extraction to guide LLMs toward patches that respect domain-specific conventions.

ClaudeLike demonstrates that general-purpose agent strategies—file editor tools and sub-agent delegation—transfer effectively to the specialized domain of security patching.

(The table also mentions Eraser, which employs custom models for specialized tasks, demonstrating yet another dimension of diversity in our ensemble.)

This diversity is precisely the point. As we discussed in our previous post on ensemble integration, no single agent can handle all bugs effectively. Different vulnerabilities require different approaches. Token constraints, project conventions, crash log quality, bug complexity—all these factors influence which strategy works best.

By building agents with diverse architectures, methodologies, and underlying models, we create an ensemble where each agent’s strengths compensate for others’ weaknesses. When Martian’s tools-first approach succeeds quickly on one bug, Prism’s patient multi-iteration strategy tackles another that requires deep exploration. When MultiRetrieval’s augmented context helps one case, Vincent’s property analysis catches project-specific requirements that others miss.

The result is a robust patching system that maintains consistent quality across the wide variety of vulnerabilities encountered in real-world software. That’s the power of diversity in ensemble-based automated patching.
